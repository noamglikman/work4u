# Work4U — Features & Use Cases (פיצ'רים ותרחישי שימוש)

> רשימה מלאה ומעודכנת, מאומתת מול המימוש בפועל. גרסת Word: docs/Work4U_Features_and_Use_Cases.docx

## על המסמך

מסמך זה מרכז את הרשימה המלאה והמעודכנת של כלל ה-Features (תרחישי שימוש) של מערכת Work4U, ומתאר עבור כל אחד מהם מה קורה בפועל. הרשימה אומתה מול המימוש האמיתי (פונקציות Lambda בצד השרת, מסכי ה-Frontend ותבנית התשתית infrastructure/template-prod.yaml) ולכן היא משקפת את המוצר כפי שמומש.

בהתאם לדרישת סעיף 05, הרשימה כוללת את כל הפיצ'רים של הפרויקט ולא רק את אלה שהופיעו בהצעה (סעיף 01). כל Feature ממוספר, ולצד כל אחד מצוין 'מימוש בקוד' שממפה אותו לקבצים ולנקודות הקצה (Endpoints) הרלוונטיים.

## פיצ'רים מתיאור הפרויקט (הצעה — סעיף 01)

### Feature 01 — הזדהות, הרשמה וניהול הרשאות

משתמש חדש נדרש ליצור חשבון כדי לגשת למערכת, תוך הפרדה בין משתמש רגיל למנהל מערכת. האימות מתבצע מול Amazon Cognito, ותפקיד המשתמש נקבע לפי קבוצת ההרשאה שלו.

**מה קורה בפועל:**

1. המשתמש מזין כתובת אימייל וסיסמה במסך ההרשמה, או מתחבר אם הוא כבר רשום.
2. המערכת שולחת בקשת אימות לשירות ניהול המשתמשים Amazon Cognito (User Pool).
3. בהרשמה, טריגר PreSignUp (AutoConfirmSignUp) מאשר את החשבון אוטומטית בסביבת ההדגמה.
4. לאחר אימות הנתונים, Cognito מנפיק אסימון אבטחה (JWT) שנשמר בלקוח ומצורף לכל בקשה עתידית.
5. תפקיד המשתמש (Users או Admins) נקבע מקבוצות ה-Cognito המשוקפות ב-claims של ה-JWT.
6. המשתמש מועבר למסך הבית. למנהל מתווספים בסרגל העליון כפתורי ניהול נוספים (הוספת מתחם, ניהול מקומות).

**מימוש בקוד:** backend/lambdas/auth_triggers/app.py · Cognito (template-prod.yaml) · frontend AuthContext.tsx, Login.tsx, Signup.tsx.

### Feature 02 — פרופיל והעדפות עבודה מותאמות אישית

לכל משתמש פרופיל הכולל העדפות עבודה: סביבה שקטה, צורך בשקעי חשמל, אינטרנט מהיר, סוג ישיבה מועדף וטווח מחירים. המערכת משתמשת במידע זה כדי להתאים ולדרג תוצאות באופן אוטומטי בכל חיפוש.

הערת דיוק: איתותי ההתאמה שמשתתפים בפועל בחישוב הציון בשרת הם מחיר, איכות Wi-Fi, סביבה שקטה ושקעים. סוג הישיבה נשמר בפרופיל ומוצג למשתמש, אך כרגע אינו משתתף בחישוב ההתאמה בצד השרת.

**מה קורה בפועל:**

1. המשתמש נכנס לאזור 'העדפות עבודה' ומסמן את הדרישות וההעדפות שלו.
2. הנתונים נשלחים בבקשת POST /preferences דרך API Gateway אל פונקציית Lambda.
3. המערכת שומרת את ההעדפות ב-Amazon DynamoDB (טבלת Work4U_UserPreferences) תחת רשומת המשתמש.
4. בכל חיפוש עתידי, ה-Frontend מזריק את ההעדפות כפרמטרים, ופונקציית Venues משתמשת בהן כאיתותי דירוג (match score).
5. התוצאות ממוינות כך שמתחמים התואמים את ההעדפות מקבלים ציון גבוה יותר ומופיעים ראשונים.

**מימוש בקוד:** backend/lambdas/preferences/app.py · calculate_match_score ב-venues/app.py · frontend Preferences.tsx, usePreferences.ts.

### Feature 03 — חיפוש והתאמת מתחמים מבוססת מיקום

המשתמש מחפש מקום עבודה קרוב המותאם לצרכיו, תוך שימוש במנגנון ההתאמה החכם ובסינון לפי רדיוס ממיקומו.

**מה קורה בפועל:**

1. האפליקציה מבקשת הרשאת מיקום (GPS) ושולחת את הקואורדינטות ל-API. כשל או דחיית הרשאה אינם חוסמים — נעשית נפילה למיקום ברירת מחדל.
2. פונקציית Venues מסננת את המתחמים לפי רדיוס חיפוש (radiusKm) סביב מיקום המשתמש, בעזרת חישוב מרחק Haversine.
3. הפונקציה שולפת מ-DynamoDB את כל מתחמי העבודה הפעילים (isActive) שברדיוס.
4. המערכת מצליבה את נתוני המתחמים עם פרופיל ההעדפות (Feature 02) ועם פרופיל הלמידה (Feature 07) לחישוב ציון התאמה.
5. מוחזרת רשימה ממוינת לפי ציון התאמה ומרחק, והמתחמים מוצגים למשתמש במפה אינטראקטיבית וברשימה.

**מימוש בקוד:** backend/lambdas/venues/app.py (get_all_venues, apply_filters_and_rank, haversine_km) · frontend Home.tsx, MapCanvas.tsx, useGeolocation.ts, useVenues.ts.

### Feature 04 — פרופיל מתחם עבודה

כל מתחם כולל מידע רלוונטי לעבודה מרחוק: איכות Wi-Fi, זמינות שקעים, רמת רעש, שעות פעילות, מחירים, תגיות מאפיינים וגרף תחזית עומס לפי שעה. המידע מוצג בצורה ברורה כדי שהמשתמש יבין מראש אם המקום מתאים לצרכיו.

הערת דיוק: 'איכות Wi-Fi' היא ערך קטגורי (נמוך / בינוני / גבוה) ולא מהירות נמדדת, ו'שקעים' הוא חיווי זמינות (hasPowerOutlets) ולא ספירת שקעים בפועל.

**מה קורה בפועל:**

1. המשתמש לוחץ על מתחם שהוצג בתוצאות החיפוש (במפה או ברשימה).
2. נשלחת בקשת GET /venues/{venueId} למשיכת כלל המידע על המתחם, כולל תמונות מ-Amazon S3.
3. מוצג מסך פרופיל מתחם הכולל: דירוג כולל, תגית עומס נוכחי, איכות Wi-Fi, זמינות שקעים, רמת רעש, שעות פעילות ומחירים.
4. הנתונים מוצגים בפורמט ויזואלי ברור (כולל גלריית תמונות וגרף תחזית) המאפשר קבלת החלטה מושכלת טרם ההגעה.

**מימוש בקוד:** backend/lambdas/venues/app.py (get_venue_by_id) · Amazon S3 · frontend Venue.tsx, ForecastGraph.tsx.

### Feature 05 — מערכת דירוגים וחיווי קהילתי בזמן אמת

המשתמשים מדרגים מתחמים ומדווחים בזמן אמת על עומס נוכחי, איכות אינטרנט ורמת רעש. כל דיווח נשמר עם חותמת זמן ותאריך, ממוצע הדירוג של המתחם מחושב מחדש, והמידע המעודכן זמין מיד לכלל משתמשי הפלטפורמה. (תצוגת תחזית העומס לפי שעה מתוארת ב-Feature 06.)

הערת דיוק: רמת העומס (crowdLevel) נאספת ונשמרת בכל דיווח, אך כרגע אינה מצטברת אוטומטית לחיווי העומס המוצג; ממוצע הדירוג הציבורי מחושב מדירוגי ה-Wi-Fi והרעש.

**מה קורה בפועל:**

1. המשתמש פותח את פרופיל המתחם ולוחץ 'דרג' (או נכנס לדירוג דרך באנר ההתראה החכמה שמופיע במסך הבית).
2. בחלון הדירוג הוא בוחר רמת עומס (פנוי / סביר / עמוס מאוד) ומדרג את איכות ה-Wi-Fi ואת רמת השקט (1–5).
3. הנתונים נשלחים בבקשת POST /ratings יחד עם חותמת זמן (createdAt) ותאריך (ratingDate).
4. המערכת שומרת את הדירוג ב-DynamoDB (טבלת Work4U_Ratings) ומחשבת מחדש את averageRating ו-ratingCount של המתחם.
5. הדירוג המעודכן זמין מיד לכל הצופים במתחם. נאכף כלל: דירוג אחד למתחם ליום לכל משתמש.

**מימוש בקוד:** backend/lambdas/ratings/app.py (submit_rating, recalculate_venue_rating) · frontend RatingModal.tsx, PushCard.tsx.

### Feature 06 — תצוגת תחזית עומסים

במסך פרופיל המתחם מוצג גרף תחזית עומס לפי שעות היום, שנועד לעזור למשתמש להעריך את העומס הצפוי לפני הגעתו.

מצב נוכחי: התחזית מוצגת מתוך שדה forecast השמור על רשומת המתחם; אם אין שדה כזה, השרת מחזיר תחזית ברירת מחדל (09:00 פנוי, 12:00 סביר, 18:00 עמוס). ניתוח אלגוריתמי של הדירוגים הקהילתיים לפי יום ושעה (Feature 05) לבניית תחזית דינמית הוא הרחבה מתוכננת ואינו ממומש בשלב זה.

**מה קורה בפועל:**

1. המשתמש נכנס למסך פרופיל המתחם (כחלק מ-Feature 04).
2. השרת מצרף לרשומת המתחם את שדה forecast, או תחזית ברירת מחדל אם השדה חסר.
3. ה-Frontend מצייר גרף/חיווי ויזואלי של רמת העומס הצפויה לפי שעה.
4. (הרחבה עתידית) אלגוריתם ינתח את דיווחי העבר לפי יום ושעה ויחליף את ברירת המחדל בתחזית מבוססת נתונים.

**מימוש בקוד:** backend/lambdas/venues/app.py (get_venue_by_id, build_default_forecast) · frontend ForecastGraph.tsx.

### Feature 07 — המלצות חכמות ממנוע למידה

המערכת מציעה למשתמש מתחמים המותאמים לטעמו, על בסיס היסטוריית השימוש שלו: חיפושים, פתיחת מתחמים ודירוגים. מנוע הלמידה צובר ניקוד העדפה לכל קטגוריית מקום ומשתפר עם הצטברות הנתונים.

הערת דיוק: המנגנון הוא ניקוד היוריסטי של זיקה לקטגוריות (categoryScores), ולא מודל למידת מכונה.

**מה קורה בפועל:**

1. פעולות המשתמש (search / open_venue / rating) נשלחות בבקשת POST /learning/event.
2. פונקציית Learning מעדכנת פרופיל ניקוד קטגוריות ומונחי חיפוש בטבלת Work4U_UserLearning.
3. בכל טעינת מתחמים, פונקציית Venues קוראת את פרופיל הלמידה ומוסיפה learningScore לציון ההתאמה.
4. רשימת 'מומלצים עבורך' מוצגת בראש סרגל הצד במסך הבית, ממוינת לפי ההתאמה.
5. הבחירות החדשות של המשתמש מזינות חזרה את הניקוד, כך שההמלצות מתחדדות לאורך זמן.

**מימוש בקוד:** backend/lambdas/learning/app.py · calculate_learning_score ב-venues/app.py · UserLearning table (template-prod.yaml).

## פיצ'רים נוספים שמומשו במערכת

### Feature 08 — ניהול מתחמים (מנהל): הוספה, עריכה ומחיקה רכה

למנהל ממשק לניהול קטלוג המתחמים: הוספת מתחם חדש, עריכת פרטים תפעוליים, ומחיקה רכה. הפעולות נגישות רק לקבוצת Admins.

**מה קורה בפועל:**

1. המנהל לוחץ 'הוספת מתחם' או 'ניהול מקומות' בסרגל העליון (מוצגים רק למנהל).
2. הוספה: מילוי טופס עם שדות חובה (שם, כתובת, שעות פעילות) ופרטים נוספים, ושליחת POST /admin/venues.
3. עריכה: עדכון שעות, מחיר, איכות Wi-Fi, רעש, שקעים, תיאור, פרטי קשר ותמונות בבקשת PUT /admin/venues/{venueId} (שם, כתובת ומיקום נעולים לעריכה).
4. מחיקה רכה: DELETE /admin/venues/{venueId} מסמן isActive=false ומסיר את המתחם מתצוגת המשתמשים בלי מחיקה פיזית.
5. הרשאת המנהל נאכפת בצד השרת לפי claims של Cognito, ולא רק באמצעות הסתרת כפתורים בלקוח.

**מימוש בקוד:** backend/lambdas/admin/app.py (create_venue, update_venue, disable_venue) · frontend AdminDialog.tsx, AdminVenuesDialog.tsx.

### Feature 09 — העלאת תמונות מתחם ל-S3 דרך Presigned URL

העלאת תמונות מתחם מתבצעת ישירות מהדפדפן אל Amazon S3 באמצעות URL חתום קצר-מועד, בלי שהבייטים עוברים דרך Lambda.

**מה קורה בפועל:**

1. המנהל בוחר תמונות בטופס הוספת/עריכת מתחם.
2. הלקוח מבקש POST /admin/venues/{venueId}/images/upload-url.
3. פונקציית Admin מייצרת presigned PUT URL בתוקף קצר (כ-15 דקות) ומחזירה אותו יחד עם ה-URL הציבורי העתידי של התמונה.
4. הדפדפן מעלה את בייטי התמונה ישירות ל-S3 באמצעות ה-URL החתום.
5. ה-URL הציבורי נשמר על רשומת המתחם ומוצג בגלריה.

**מימוש בקוד:** backend/lambdas/admin/app.py (generate_upload_url) · Amazon S3 · frontend api/live.ts.

### Feature 10 — מועדפים (Favorites)

המשתמש יכול לסמן מתחמים כמועדפים (אייקון לב) לגישה מהירה. המועדפים נשמרים בפרופיל המשתמש.

**מה קורה בפועל:**

1. בכרטיס המתחם, המשתמש לוחץ על אייקון הלב כדי להוסיף או להסיר מהמועדפים.
2. רשימת favoriteVenueIds מתעדכנת ונשמרת עם פרופיל המשתמש ב-DynamoDB.
3. מספר המועדפים מוצג כסטטיסטיקה במסך הפרופיל האישי (Feature 13).

**מימוש בקוד:** frontend VenueListCard.tsx, types/api.ts (favoriteVenueIds) · נשמר עם UserPreferences.

### Feature 11 — היסטוריית דירוגים וניהולם

מסך 'הדירוגים שלי' מציג את כל הדיווחים שהמשתמש שלח, עם אפשרות מחיקה. עדכון דירוג קיים נתמך ב-API (עריכה מה-UI היא הרחבה עתידית).

**מה קורה בפועל:**

1. המשתמש נכנס ל'הדירוגים שלי' מהסרגל העליון.
2. נטענת רשימת הדירוגים בבקשת GET /ratings/my (שאילתה יעילה דרך ה-GSI userId-index).
3. לכל פריט מוצגים: שם המתחם, תאריך, רמת עומס, דירוג Wi-Fi /5 ורעש /5, וכפתור מחיקה.
4. מחיקה: DELETE /ratings/{ratingId} (מותרת לבעל הדירוג; מנהל יכול למחוק כל דירוג), ומפעילה חישוב מחדש של ממוצע המתחם.
5. עדכון דירוג נתמך בצד השרת בבקשת PUT /ratings/{ratingId}.

**מימוש בקוד:** backend/lambdas/ratings/app.py (get_my_ratings, delete_rating, update_rating) · frontend History.tsx, useRatings.ts.

### Feature 12 — שחזור ואיפוס סיסמה

תהליך דו-שלבי לאיפוס סיסמה מול Amazon Cognito, נגיש מקישור 'שכחתי סיסמה' במסך ההתחברות.

**מה קורה בפועל:**

1. במסך ההתחברות, המשתמש לוחץ 'שכחתי סיסמה'.
2. שלב 1: הזנת כתובת אימייל ושליחת קוד איפוס (Cognito forgotPassword).
3. שלב 2: הזנת הקוד שהתקבל יחד עם סיסמה חדשה ואישורה (confirmForgotPassword).
4. בהצלחה, המשתמש מוחזר למסך ההתחברות עם הסיסמה החדשה.

**מימוש בקוד:** frontend ForgotDialog.tsx, api/auth.ts · Amazon Cognito.

### Feature 13 — פרופיל אישי, סטטיסטיקות והתנתקות

מסך פרופיל אישי המציג את פרטי החשבון, תג מנהל לבעלי הרשאה, סטטיסטיקות שימוש, קיצורים להגדרות ויציאה מהמערכת.

**מה קורה בפועל:**

1. המשתמש נכנס לפרופיל מאייקון האווטאר בסרגל העליון.
2. מוצגים שם, אימייל, תג 'מנהל' לבעלי הרשאה, וסטטיסטיקות (דיווחים, מועדפים).
3. שורות פעולה: 'העדפות עבודה' (מעבר לעריכת Feature 02), 'התראות חכמות' ו'אודות Work4U'.
4. כפתור 'התנתקות' מבצע signOut מול Cognito ומחזיר את המשתמש למסך ההתחברות.

**מימוש בקוד:** frontend Profile.tsx, AuthContext.tsx.

### Feature 14 — ניווט למתחם וטיפול במיקום

פתיחת ניווט חיצוני למתחם, וטיפול חיני (non-blocking) בהרשאת המיקום של הדפדפן.

**מה קורה בפועל:**

1. במסך פרופיל המתחם, כפתור 'פתח בניווט' פותח את Google Maps עם יעד לקואורדינטות המתחם.
2. אם הרשאת ה-GPS נדחית או אינה זמינה (או אין HTTPS), המערכת אינה נחסמת ונופלת למיקום ברירת מחדל.
3. המשתמש יכול לבחור 'אזור חיפוש' ידני כתחליף ל-GPS.

**מימוש בקוד:** frontend Venue.tsx (navigate), useGeolocation.ts, Home.tsx (בורר אזור חיפוש), lib/searchLocations.ts.

### Feature 15 — התאמת ממשק (ערכת נושא)

פאנל הגדרות מראה המאפשר למשתמש להתאים אישית את ערכת הנושא, צבע ההדגשה והגופן של הממשק.

**מה קורה בפועל:**

1. המשתמש פותח את פאנל המראה הצף.
2. בוחר ערכת נושא, צבע הדגשה וגופן.
3. ThemeContext מעדכן את משתני ה-CSS הגלובליים והממשק מתעדכן מיידית.

**מימוש בקוד:** frontend AppearanceSettings.tsx, ThemeContext.tsx, lib/theme.ts.
